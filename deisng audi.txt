esign Audit: Midnight Automation Voyage
Overall Assessment
This is a well-structured training platform with strong content architecture (15 modular lessons, typed data layer, shadcn/ui components). The training-app has a polished dark terminal aesthetic that fits the Playwright audience. The practice-app is intentionally simple (it's a test target), which is appropriate. The biggest design opportunities are in typography readability, accessibility gaps, and training UX flow.

Critical Issues (fix these first)
1. Monospace body text kills reading speed — JetBrains Mono is set on the root <div> in App.tsx (line 31) as the font for all UI text. Your index.css loads Inter and sets it on <body>, but the inline style overrides it. Research (Arditi & Cho, 2005; Bernard et al., 2003) shows monospace fonts reduce reading speed 10-15% vs. proportional fonts for prose. For a training app where users read paragraphs of instructional content, this is a meaningful comprehension hit.
→ Fix: Remove the inline fontFamily from App.tsx line 31. Let body use Inter (already loaded). Keep JetBrains Mono only on headings (h2 in LessonView line 366) and <pre><code> blocks (already scoped in index.css line 49-51).
Impact: High | Effort: 5 min
2. No keyboard navigation — No keydown listener for ArrowLeft/ArrowRight to navigate lessons. For a training app, keyboard-driven progression is essential (WCAG 2.1 SC 2.1.1). Users reading dense content shouldn't need to mouse to the bottom to advance.
→ Fix: Add useEffect in App.tsx with ArrowLeft → goPrev(), ArrowRight → goNext().
Impact: High | Effort: 15 min
3. Sidebar disappears completely when closed — if (!open) return null (Sidebar.tsx line 16) removes the entire sidebar from the DOM. There's no persistent affordance to reopen it — the only way back is the hamburger icon in LessonView, which requires visual scanning. Users who close it may lose orientation.
→ Fix: Render a collapsed rail (32px wide) with a toggle icon instead of return null. Keeps spatial consistency and provides a persistent re-open target.
Impact: High | Effort: 30 min
4. goNext() auto-marks lesson complete — App.tsx line 17-18: clicking "Next Lesson" calls markComplete(currentLesson) before advancing. If a user skips ahead to preview content, it falsely registers completion. This undermines training integrity — ref: Kirkpatrick Model Level 2 (learning verification should be intentional, not incidental).
→ Fix: Remove markComplete from goNext(). The explicit "Mark Complete" button in LessonView (line 429) is the correct completion path. Optionally, gate completion behind quiz/exercise submission.
Impact: High (UX integrity) | Effort: 10 min

Improvements (meaningful quality uplift)
5. Font sizes below readability threshold — Multiple instances of text-[9px], text-[10px], and text-[11px] across Sidebar and LessonView. Nine of these are content-bearing text (audience labels, progress counters, prompt contexts), not decorative. WCAG recommends minimum 12px for body-adjacent text; 9-10px fails on standard-density screens.
→ Fix: Floor all content-bearing text at text-[11px] minimum, ideally text-xs (12px). Keep text-[10px] only for truly auxiliary labels (language tag in code blocks).
Impact: Medium | Effort: 20 min
6. No responsive sidebar behavior — sidebarOpen defaults to true regardless of viewport. On <768px, the 288px sidebar consumes 40%+ of screen width, squeezing lesson content.
→ Fix: Initialize sidebarOpen using window.matchMedia('(min-width: 768px)').matches. Add an overlay/drawer pattern for mobile.
Impact: Medium | Effort: 45 min
7. No aria-live for progress updates — When completedLessons changes, screen readers don't announce the new progress percentage.
→ Fix: Wrap the progress section in Sidebar with aria-live="polite". Add role="progressbar" attributes.
Impact: Medium | Effort: 10 min
8. Quiz has no retry — Once submitted, the quiz locks (line 82: disabled={submitted}). Training best practice: allow retry for incorrect answers to reinforce learning (ref: retrieval practice effect — Roediger & Butler, 2011).
→ Fix: Add a "Try Again" button that resets selected and submitted state when the answer was incorrect.
Impact: Medium | Effort: 15 min
9. No visual lesson progress indicator in main content — The top bar shows "01 / 15" but there's no progress bar in the lesson view itself. Users lose context of overall journey when sidebar is closed.
→ Fix: Add a thin progress bar at the top of LessonView (similar to reading progress indicators in Medium/Substack).
Impact: Low-Medium | Effort: 20 min
10. DiffCodeBlock starter font is 11px — The side-by-side code comparison uses text-[11px] (line 146, 171). At this size, monospace characters become hard to distinguish (l vs 1, O vs 0). Code blocks should be minimum 12px.
→ Fix: Bump to text-[12px] for both panes.
Impact: Medium | Effort: 5 min

Polish (nice-to-have)
11. Practice-app DashboardPage uses inline styles (line 20: style={{ display: "flex", gap: 16 }}) while the rest uses CSS classes. Minor inconsistency.
12. Practice-app's failCounts is a module-level mutable object (LoginPage line 5) — won't survive hot reload. Fine for a test target but worth a comment.
13. Lesson icons are plain emoji strings. Consider wrapping in a consistent <span role="img" aria-label> for screen reader clarity.

What's Working Well

Content architecture is excellent — 15 typed lesson modules, clean registry pattern in data/index.ts, separation of content from presentation
Dual-track audience design — Audience badges, conditional skip paths for Developer vs. Non-Coder tracks
shadcn/ui foundation — Full component library installed, Accordion/Badge/Progress properly used
Code exercise UX — DiffCodeBlock with starter→solution comparison, hint progressive disclosure, copy-to-clipboard
Practice-app as test target — data-testid attributes throughout, realistic form validation, session auth, checkout flow. Purpose-built for the training content
Color system is cohesive — Emerald accent throughout training-app, blue accent in practice-app, consistent zinc neutrals
