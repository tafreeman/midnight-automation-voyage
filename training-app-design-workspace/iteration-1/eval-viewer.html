\n Writing Your First Playwright Test\n

\n

Navigate to pages and interact with elements using page.goto() and page.click()

\n
\n All Roles\n Hands-on\n ~15 min\n
\n
\n\n \n
\n\n \n
\n
\n

\n \n What You Will Learn\n

\n

\n Every Playwright test starts with two fundamental actions: navigating to a URL and interacting with elements on the page. In this lesson you will learn page.goto() for navigation and page.click() for triggering user interactions. By the end, you will write a working test against the practice app's login page.\n

\n
\n

\n These two methods form the foundation of every Playwright test. Master them and you can automate any user workflow.\n

\n
\n
\n
\n\n \n
\n
\n

\n \n Navigating with page.goto()\n

\n

\n page.goto(url) navigates the browser to the given URL. Playwright automatically waits for the page to reach the load state before continuing. If you configure a baseURL in your Playwright config, you can use relative paths.\n

\n
\n
\n
\n \n typescript\n
\n \n
\n 
// Navigate to an absolute URL\nawait page.goto('http://localhost:5173/login');\n\n// With baseURL configured, use a relative path\nawait page.goto('/login');\n\n// Wait for a specific load state\nawait page.goto('/dashboard', {\n  waitUntil: 'networkidle'\n});
\n
\n
\n

\n Tip: Always set baseURL in playwright.config.ts so your tests use relative paths. This makes them portable across environments.\n

\n
\n
\n
\n\n \n
\n
\n

\n \n goto() Options Reference\n

\n
\n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n

waitUntil'load'When to consider navigation succeeded: 'load', 'domcontentloaded', 'networkidle', 'commit'

\n \n Clicking Elements with page.click()\n

\n

\n page.click(selector) finds an element matching the selector, scrolls it into view, and clicks it. Playwright automatically waits for the element to be visible, enabled, and stable before clicking. The preferred selectors are data-testid attributes and role-based locators.\n

\n
\n
\n
\n \n typescript\n
\n \n
\n 
// Click by data-testid (recommended)\nawait page.click('[data-testid=\"login-button\"]');\n\n// Modern locator API (preferred)\nawait page.getByTestId('login-button').click();\n\n// Click by role (great for accessibility)\nawait page.getByRole('button', { name: 'Log In' }).click();\n\n// Click by visible text\nawait page.getByText('Submit').click();
\n
\n
\n

\n Warning: Avoid CSS selectors like .btn-primary or #root > div > button. These break when the UI is refactored. Always prefer data-testid or role-based locators.\n

\n
\n
\n
\n\n \n
\n
\n

\n \n Putting It Together: A Complete Login Test\n

\n

\n Combining page.goto(), page.fill(), and page.click() lets you automate a complete login flow. Notice the Arrange / Act / Assert structure.\n

\n
\n
\n
\n \n typescript\n Complete login test\n
\n \n
\n 
import { test, expect } from '@playwright/test';\n\ntest('successful login redirects to dashboard', async ({ page }) => {\n  // ARRANGE - navigate to the login page\n  await page.goto('/login');\n\n  // ACT - fill in credentials and submit\n  await page.getByTestId('email-input').fill('user@test.com');\n  await page.getByTestId('password-input').fill('Password123!');\n  await page.getByTestId('login-button').click();\n\n  // ASSERT - verify we landed on the dashboard\n  await expect(page).toHaveURL(/dashboard/);\n});
\n
\n
\n

\n Tip: Notice the Arrange / Act / Assert pattern. Every well-structured test follows this. When reviewing tests, check that each section is present.\n
