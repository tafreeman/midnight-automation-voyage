Decouple goNext() from Auto-Complete
High Impact 10 min
File: src/App.tsx Lines: 15-20 ADR: ADR-003 Audit: UI-4
Problem

goNext() calls markComplete(currentLesson) on every forward navigation. Learners who simply preview later lessons get false "complete" marks. Violates Kirkpatrick Level 2: completion should be an intentional act.

Change
// App.tsx lines 15-20 — BEFORE
const goNext = () => {
  if (currentLesson < lessons.length - 1) {
    markComplete(currentLesson);
    setCurrentLesson((p) => p + 1);
  }
};

// AFTER — navigation only, no side effects
const goNext = () => {
  if (currentLesson < lessons.length - 1) {
    setCurrentLesson((p) => p + 1);
  }
};

// markComplete() is still called from the explicit
// "Mark Complete" button in LessonView (no change needed there)
Verification
  • Click "Next" → lesson advances but completion dot does NOT appear
  • Click "Mark Complete" button → completion dot appears correctly
  • Progress counter only increments on explicit mark, not navigation
3
Add localStorage Persistence for Progress
High Impact 30 min
File: src/App.tsx Depends on: Task #2 ADR: ADR-002 Audit: TP-1
Problem

All progress is lost on page refresh. completedLessons and currentLesson are ephemeral useState values. For a 15-module course, learners cannot resume multi-session learning.

Design Decisions
Key trade-off: Versioned storage key enables future schema migration. Using mav-progress-v1 as the key means we can bump to v2 later and write a migration function without breaking existing learners.
Change
// App.tsx — add persistence helpers above the component

const STORAGE_KEY = "mav-progress-v1";

interface PersistedState {
  currentLesson: number;
  completedLessons: number[];
}

function loadProgress(): PersistedState | null {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    return raw ? JSON.parse(raw) : null;
  } catch {
    return null;
  }
}

function saveProgress(current: number, completed: Set<number>) {
  localStorage.setItem(STORAGE_KEY, JSON.stringify({
    currentLesson: current,
    completedLessons: [...completed],
  }));
}

// Inside component — hydrate initial state
const saved = loadProgress();
const [currentLesson, setCurrentLesson] = useState(saved?.currentLesson ?? 0);
const [completedLessons, setCompletedLessons] = useState<Set<number>>(
  new Set(saved?.completedLessons ?? [])
);

// Add useEffect to persist on change
useEffect(() => {
  saveProgress(currentLesson, completedLessons);
}, [currentLesson, completedLessons]);
Verification
  • Complete 3 lessons → refresh page → completion dots persist
  • Navigate to lesson 10 → refresh → opens on lesson 10
  • Clear localStorage → refresh → starts fresh at lesson 0
  • Corrupted localStorage value → gracefully falls back to defaults

    // Don't capture if user is typing in an input/textarea

URL Routing (Hash-Based Deep Linking)
Medium Impact 1-2 hrs
File: src/App.tsx Depends on: Task #3 ADR: ADR-008 Audit: TP-4
Design Decision
Hash-based routing recommended over React Router because: (1) no new dependency needed, (2) works with static file hosting, (3) simple enough for a flat lesson list, (4) react-router-dom is not in training-app's dependencies. Format: #lesson/9
Change
// App.tsx — sync URL hash ↔ currentLesson state

// Read hash on mount
function getLessonFromHash(): number | null {
  const match = window.location.hash.match(/^#lesson\/(\d+)$/);
  if (!match) return null;
  const idx = parseInt(match[1], 10);
  return idx >= 0 && idx < lessons.length ? idx : null;
}

// Priority: URL hash > localStorage > default (0)
const initialLesson = getLessonFromHash() ?? saved?.currentLesson ?? 0;

// Update hash when lesson changes
useEffect(() => {
  window.location.hash = `lesson/${currentLesson}`;
}, [currentLesson]);

// Listen for browser back/forward
useEffect(() => {
  const onHashChange = () => {
    const idx = getLessonFromHash();
    if (idx !== null) setCurrentLesson(idx);
  };
  window.addEventListener("hashchange", onHashChange);
  return () => window.removeEventListener("hashchange", onHashChange);
}, []);
Verification
  • Navigate to #lesson/9 → opens Module 10 (0-indexed)
  • Click sidebar items → URL hash updates
  • Browser back button → returns to previous lesson
  • Invalid hash (#lesson/99) → falls back to localStorage or 0
  • Share URL with colleague → opens correct lesson

Role Selector with Filtered Learning Paths
Medium Impact 1-2 hrs
File: src/App.tsx, src/components/Sidebar.tsx Depends on: Task #9 Audit: TP-5
Approach

Add a role toggle (All / Developer / Non-Coder) to the sidebar header. When a role is selected, lessons not tagged for that role are visually dimmed but still accessible (not hidden).

// App.tsx — new state
type RoleFilter = "all" | "developer" | "non-coder";
const [roleFilter, setRoleFilter] = useState<RoleFilter>("all");

// Sidebar.tsx — filter logic (dim, don't hide)
const isRelevant = (lesson: Lesson) => {
  if (roleFilter === "all") return true;
  if (!lesson.audience) return true; // "All Roles"
  return lesson.audience.toLowerCase().includes(roleFilter);
};
Design Note

Dim, don't hide — hiding lessons would break the lesson index numbering and confuse learners who hear about "Module 9" from a colleague. Instead, off-role lessons get opacity-40 and a small badge indicating they're optional for the selected role.
